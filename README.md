# SmartMatching

`SmartMatching` сравнивает две генеалогии в формате `import.json` и возвращает все совпадения, прошедшие порог, в порядке от лучшего к худшему.

## Что считается входом

- `tree1` и `tree2` — это **два отдельных дерева**, каждое в формате JSON-массива персон (структура как в `import.json`).
- Поле `treeId` у всех персон внутри одного входного массива должно быть одинаковым.
- Вход можно передавать как строку JSON или как уже распарсенный `list[dict]`.

## Режимы сравнения

Алгоритм поддерживает переключаемый режим `exact_match`:

- `exact_match=true` (по умолчанию): строгая фильтрация кандидатов перед расчетом score.
- `exact_match=false`: обычный fuzzy-режим (без строгого предварительного отсечения).

### Логика exact match

Для пары персон обязательно:

- непустые поля в обеих записях: `surname`, `name`, `middleName`, `birthdate`, `birthplace`;
- строгое совпадение (с нормализацией) `surname`, `name`, `middleName`;
- совпадение `birthdate` по правилу частичной точности;
- `maidenName` сравнивается строго только если присутствует в обеих записях;
- сравнение `birthplace` остается AS-IS (текущий алгоритм `place_similarity`).

Если условия exact match не выполнены, пара сразу отбрасывается.

### Правила exact match для даты рождения

- сравнение проводится только если в обеих датах есть хотя бы год;
- если в одной записи только год — сравнивается только год;
- если в одной записи есть год+месяц — сравниваются год и месяц;
- если обе записи содержат день+месяц+год — требуется полное совпадение даты.

## Что возвращает алгоритм

Возвращается объект:

- `tree1Id` — id первого дерева;
- `tree2Id` — id второго дерева;
- `tree1Size` — размер первого дерева;
- `tree2Size` — размер второго дерева;
- `matchesCount` — число найденных совпадений;
- `matches` — список мэтчей, отсортированный по `score` по убыванию.

Каждый элемент `matches`:

- `score`
- `tree1.id`, `tree1.personId`, `tree1.person`
- `tree2.id`, `tree2.personId`, `tree2.person`

## Тестирование гипотезы

### 1) Требования к тестовой выборке

- **Регионы:** Тверская область и Ярославская область.
- **Критерий отнесения дерева к региону:** дерево относится к региону, если более 50% персон имеют связь с этим регионом по месту рождения.
- **Правило определения региона по месту рождения:** подстрока `Твер` -> Тверской регион, подстрока `Ярослав` -> Ярославский регион.
- **Минимальный размер дерева:** в тест включаются только деревья с числом персон `>= 50`.
- **Ограничение по количеству:** используется вся доступная выборка деревьев, удовлетворяющая критериям, без ограничения по количеству.
- **Цель отбора:** максимизация вероятности пересечений при сохранении вычислительной реализуемости.

### 2) Инструкция по запуску контейнера (только exact-режим)

Сборка образа:

```bash
docker build -t smartmatching:latest .
```

Запуск сравнения:

```bash
docker run --rm -v "$(pwd):/data" smartmatching:latest --tree1 /data/tree1.json --tree2 /data/tree2.json --threshold 90 --exact-match true
```

### 3) Требования к логгированию результатов

- В каждом запуске необходимо сохранять полный JSON-ответ алгоритма.
- Необходимо сохранять результаты **всех** сравнений деревьев, выполненных в рамках теста.
- Минимально фиксируемые поля ответа: `tree1Id`, `tree2Id`, `tree1Size`, `tree2Size`, `matchesCount`, `matches`.

Пример структуры возвращаемого JSON:

```json
{
  "tree1Id": "tree_1_id",
  "tree2Id": "tree_2_id",
  "tree1Size": 134,
  "tree2Size": 128,
  "matchesCount": 3,
  "matches": [
    {
      "score": 97.6,
      "tree1": {
        "id": "tree_1_id",
        "personId": "person_1_id",
        "person": { "...": "..." }
      },
      "tree2": {
        "id": "tree_2_id",
        "personId": "person_2_id",
        "person": { "...": "..." }
      }
    }
  ]
}
```

## Методология сравнения

Сравнение выполняется по всем парам персон из двух деревьев (`N x M`).  
Для каждой пары рассчитывается итоговый `score` как взвешенное среднее по полям.

| Поле | Как сравнивается | Балл при отсутствии данных | Вес | Особенности |
|---|---|---:|---:|---|
| `surname` (фамилия) | exact=true: строгое совпадение, exact=false: `rapidfuzz.token_sort_ratio` | 70 | 0.28 | В exact=true пустое/несовпадение отсекает пару |
| `name` (имя) | exact=true: строгое совпадение, exact=false: `rapidfuzz.token_sort_ratio` | 70 | 0.24 | В exact=true пустое/несовпадение отсекает пару |
| `middleName` (отчество) | exact=true: строгое совпадение, exact=false: `rapidfuzz.token_sort_ratio` | 70 | 0.10 | В exact=true пустое/несовпадение отсекает пару |
| `birthdate` (дата рождения) | exact=true: сравнение по частичной точности, exact=false: сравнение диапазонов | 50 | 0.28 | В exact=true обе даты должны содержать хотя бы год |
| `birthplace` (место рождения) | Токенизация + триграммы + коэффициент Жаккара | 50 | 0.10 | Удаляются служебные топонимы (`г`, `обл`, `район` и т.д.) |
| `maidenName` (девичья фамилия) | exact=true: строгое совпадение, exact=false: `rapidfuzz.token_sort_ratio` | 70 | 0.12 | Учитывается **только если есть в обеих** записях |

## Логика выбора лучшего совпадения

1. Для каждой пары персон между деревьями считается `score`.
2. Пары ниже `threshold` отбрасываются.
3. Все пары с `score >= threshold` попадают в результат.
4. Список `matches` сортируется по убыванию `score`.

## Локальный запуск

Точка входа — `main.py`:

```bash
python3 main.py --tree1 ./import.json --tree2 ./import.json
```

С порогом:

```bash
python3 main.py --tree1 ./import.json --tree2 ./import.json --threshold 90
```

С отключением exact-режима:

```bash
python3 main.py --tree1 ./import.json --tree2 ./import.json --threshold 90 --exact-match false
```

## Docker

Сборка образа:

```bash
docker build -t smartmatching:latest .
```

Запуск контейнера:

```bash
docker run --rm -v "$(pwd):/data" smartmatching:latest --tree1 /data/import.json --tree2 /data/import.json --threshold 90
```

Пояснение:
- `ENTRYPOINT` контейнера: `["python", "main.py"]`;
- аргументы после имени образа передаются в `main.py`;
- через `-v "$(pwd):/data"` входные файлы из текущей директории доступны внутри контейнера по пути `/data`.
